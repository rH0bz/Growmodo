# Growmodo — Custom WordPress Block Theme + Style Guide

A fully custom **Gutenberg block theme** for the Growmodo real-estate design ("Estatein" is the
placeholder brand name from the design export), plus a **visual style guide** generated from the
same design architecture. Built AI-first — no page builder, no plugin stack, everything in code
and under version control.

```
Growmodo/          ← the WordPress theme (source only — build it with npm, see below)
style-guide/       ← self-contained visual component catalog (open index.html in a browser)
README.md
```

---

## 1. How this theme and style guide were built

The build follows a two-stage pipeline: **document the architecture first, then generate the code
from that documentation** — never straight from the raw design files.

**Step 1 — Ingest (design → architecture).** The raw design export (6 HTML pages from a
Figma-style export: Home, About, Contact, Services, Properties archive, Property single) was
analyzed page by page. Every page was decomposed into a reusable hierarchy — **Templates →
Sections → Modules → Components** — plus the shared pieces: design tokens (colors, typography,
buttons), globals (header/footer) and behaviors. That decomposition lives in an architecture wiki
as the **Source of Truth (SOT)**: framework-independent documentation the code is generated from.

**Step 2 — Build (architecture → WordPress theme).** From the SOT, the block theme was generated:

- **26 custom dynamic blocks** (2 globals + 24 sections) across **8 templates**
- A design-token system: one `tailwind.config.js` + one brand CSS layer holds every color, font
  size, radius and shadow — restyle the whole site from one file
- A **`property` custom post type** with native meta boxes (price, beds/baths, area, build year,
  key features, fee breakdown) and **real 5-facet archive filtering** (location, type, price,
  size, build year) — no plugin involved
- Native lead-form handling (contact / property lead / property inquiry) via WordPress core APIs

**Step 3 — Verify.** The build was audited against the original design (assets, sizing, layout
structure, missing content) and refined until faithful.

**The style guide** was rendered from the same SOT as one self-contained HTML page, styled in the
site's own branding, reusing the theme's real token set and assets — so design docs and code can
never drift apart.

**Build effort: ~3 hours total** — roughly **2 hours for the initial build** and **~1 hour of
follow-up updates**. **Zero manual coding**: the entire theme was produced through AI prompting,
directed by an experienced developer. The know-how lives in the documented architecture and the
prompts — not in hand-typed code.

## 2. How to use the theme

1. **Copy the theme folder** into your WordPress themes directory, using the lowercase slug
   `growmodo`:

   ```
   wp-content/themes/growmodo/          ← contents of Growmodo/ from this repo
   ```

   (e.g. on XAMPP: `C:\xampp\htdocs\<your-site>\wp-content\themes\growmodo\`)

2. **Install and build the assets** (requires [Node.js](https://nodejs.org/)). The repo ships
   source only — the compiled CSS/JS that WordPress actually loads is generated by:

   ```
   cd wp-content/themes/growmodo
   npm install
   npm run build
   ```

   > Windows/PowerShell note: if you get a "running scripts is disabled" error, use `npm.cmd
   > install` and `npm.cmd run build` instead.

   `npm install` is only needed once per copy; re-run `npm run build` after any source change
   (or use `npm run start` for watch mode while developing).

3. **Activate** the theme in wp-admin → Appearance → Themes → *Growmodo*.

4. **Add content**: a **Properties** menu appears in wp-admin (the custom post type). Create
   property posts with a featured image, gallery, price and details — Home's "Featured
   Properties" and the Properties archive (with its filters) are live queries, so they populate
   automatically.

## 3. What is the style guide?

Open `style-guide/index.html` in a browser (keep it next to the `Growmodo/` folder — it reuses
the theme's icons and images).

It's a **visual component catalog / brand guide**: one page showing the complete design system —
color tokens, the type scale, radii and the signature "outer glow" device, every button and form
style, all 6 atomic components, 15 modules, live previews of all 24 page sections, the global
header/footer, and the site behaviors. Think of it as the contract between design and code:

- **For developers** — the reference for what every piece should look like before touching code.
- **For editors/stakeholders** — a browsable overview of every building block the site has,
  without needing a WordPress install.

It has no build step and no dependencies to install — it's a single HTML file.

## 4. Why we build this way

### Zero plugins (or as close as possible)

Everything a typical WordPress agency reaches for a plugin to do, this theme does natively:
custom blocks instead of a page builder, `register_post_type` + hand-built meta boxes instead of
ACF/CPT-UI, core form handling instead of a forms plugin.

**Why it matters — from experience maintaining plugin-heavy sites:**

- **Maintenance treadmill.** Every plugin is another thing to update, and every update is a
  chance for something to break or conflict with another plugin. Sites with 20–30 plugins turn
  routine maintenance into a recurring project of its own.
- **Security.** Outdated and abandoned plugins are the single most common way WordPress sites
  get hacked. Every plugin you add is third-party code running with full access to your site —
  if it isn't maintained properly, it's an open door. Fewer plugins = a dramatically smaller
  attack surface.
- **You don't own the behavior.** When a plugin changes direction, goes paid, or gets abandoned,
  your site's features are hostage to it. Here, every feature is code in this repo that any
  developer can read and change.

### Faster than builder-dependent sites

Page builders (Elementor, Divi, WPBakery…) load megabytes of generic CSS/JS on every page to
support every feature they *might* need. This theme ships only what the design actually uses:
Tailwind compiles a single small stylesheet from exactly the markup in the theme, and blocks
render server-side as plain HTML. The result is faster page loads and far better Core Web Vitals
than a builder can realistically achieve — with no caching-plugin gymnastics required.

### Flexible, git-native, multi-developer friendly

- **Everything is code, so everything is version-controlled**: diffs, code review, branches,
  safe rollbacks, and a real history of every change. A builder stores your layout as opaque
  data in the database — you can't diff it, review it, or meaningfully merge it.
- **No lock-in**: the design system lives in the architecture documentation and one token file
  (`tailwind.config.js` + the brand CSS layer). Rebranding or restyling is a small, reviewable
  change — not a click-through of hundreds of builder screens.
- **Team-ready**: multiple developers can work in parallel on separate blocks/branches and merge
  through git like any modern software project. Any developer with WordPress + Node.js can clone,
  build, and contribute — no builder licenses, no exported/imported templates, no "it only works
  on the site where it was clicked together."
